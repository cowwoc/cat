# Automate git-filter-repo Upstream Release Detection

## Goal
Add automation to monitor the upstream git-filter-repo repository for new releases and automatically
trigger the binary rebuild workflow in `.github/workflows/build-git-filter-repo.yml`, eliminating
the need for manual `workflow_dispatch` to pick up new versions.

## Background
The current `build-git-filter-repo.yml` workflow only runs on manual `workflow_dispatch` trigger.
When git-filter-repo releases a new version upstream, maintainers must manually detect the release
and trigger the workflow. This creates a maintenance burden and risks running outdated binaries.

## Changes Required

1. Add a scheduled trigger to `check-git-filter-repo-upstream.yml` that checks the upstream
   `newren/git-filter-repo` repository for new releases via the GitHub API.
2. Add a `check-upstream` job that:
   - Fetches the latest release tag from `https://api.github.com/repos/newren/git-filter-repo/releases/latest`
   - Compares it to the last-built version in `plugin/.git-filter-repo-config/release.conf`
   - Updates `release.conf` with the new `RELEASE_TAG` and `SOURCE_SHA256` when a new version is found
   - Triggers `build-git-filter-repo.yml` via `workflow_dispatch` API when a new version is available
3. Update `build-git-filter-repo.yml` comments to document the automated trigger mechanism.

## Research Findings

### Version tracking
The current built version is stored in `plugin/.git-filter-repo-config/release.conf` as
`RELEASE_TAG="git-filter-repo-vX.Y.Z"`. The upstream latest release is retrieved via the GitHub
API: `GET https://api.github.com/repos/newren/git-filter-repo/releases/latest`, which returns a
JSON object with a `"tag_name"` field (e.g., `"v2.38.0"`).

### Triggering the build workflow
GitHub Actions restricts workflows triggered by `GITHUB_TOKEN` from triggering other workflows via
`push` events to prevent infinite loops. Three options were considered:

**Option A — Tag push (rejected)**: Push a tag `git-filter-repo-vX.Y.Z` to trigger the existing
`push: tags: 'git-filter-repo-v*'` trigger in `build-git-filter-repo.yml`. Rejected because GitHub
states that `push` events generated by `GITHUB_TOKEN` will not trigger new workflow runs.

**Option B — PAT secret (rejected)**: Use a Personal Access Token stored as a repository secret
to push the tag. This bypasses the `GITHUB_TOKEN` restriction but introduces operational burden:
PATs expire, require manual rotation, and carry broader permissions than needed.

**Option C — `gh workflow run` (chosen)**: Call the GitHub Actions `workflow_dispatch` REST API via
`gh workflow run build-git-filter-repo.yml -f version=vX.Y.Z`. GitHub docs state that
`workflow_dispatch` events triggered via the API are NOT subject to the `GITHUB_TOKEN` restriction —
they can trigger other workflows. This requires only `actions: write` permission on the check
workflow's token.

### SOURCE_SHA256 update requirement
The `build-git-filter-repo.yml` workflow reads `SOURCE_SHA256` from `release.conf` to verify the
upstream source before building. When a new version is detected, the check workflow must:
1. Download `git-filter-repo.py` from the new upstream tag URL
2. Compute its SHA256
3. Update `release.conf` with the new `RELEASE_TAG` and `SOURCE_SHA256`
4. Commit and push this change before triggering the build workflow

The build workflow triggered via `workflow_dispatch` with `--ref` will check out the latest commit
on the ref, which includes the updated `release.conf`.

### PLATFORM_SHA256 values
After the build workflow creates the new release with platform binaries, the `PLATFORM_SHA256_*`
values in `release.conf` remain stale (referring to the previous version) until manually updated.
Updating these values is outside the scope of this issue — the automation here covers detecting new
versions and triggering the build, not completing the full release cycle.

### Action pinning convention
All existing workflows pin actions to exact commit SHAs with version comments. The new workflow
must follow the same convention. The SHA for `actions/checkout v4.3.1` (already used in the repo)
is `34e114876b0b11c390a56381ad16ebd13914f8d5`.

### Error handling edge cases
- **No releases found**: Validate HTTP response; fail with clear message if `tag_name` is missing.
- **Invalid tag format**: Validate against `^v[0-9]+\.[0-9]+\.[0-9]+$`; fail fast on mismatch.
- **Source download failure**: Use `curl -fsSL --max-time 60`; fail with error if download fails.
- **Already up to date**: Set `update_needed=false`, log informational message, exit cleanly.

## Post-conditions

- [ ] A scheduled/automated trigger exists to detect new upstream git-filter-repo releases
- [ ] The build workflow runs automatically when a new upstream release is detected
- [ ] No manual `workflow_dispatch` required for routine version updates
- [ ] The mechanism is documented in `build-git-filter-repo.yml` comments
- [ ] No regressions introduced to existing manual trigger support

## Jobs

### Job 1

- Create `.github/workflows/check-git-filter-repo-upstream.yml` with the following structure:
  - License header: hash-comment style (same as `build-git-filter-repo.yml`)
  - Workflow name: `Check git-filter-repo Upstream Release`
  - Triggers: `schedule: - cron: '0 6 * * *'` (daily at 06:00 UTC) and `workflow_dispatch`
  - Permissions at workflow level: `contents: write`, `actions: write`
  - Job `check-upstream` on `ubuntu-latest`:
    - Step 1 — Checkout: `uses: actions/checkout@34e114876b0b11c390a56381ad16ebd13914f8d5 # v4.3.1`
    - Step 2 — Get latest upstream release (id: upstream):
      ```bash
      set -euo pipefail
      RESPONSE=$(curl -fsSL \
        -H "Accept: application/vnd.github+json" \
        -H "Authorization: Bearer ${{ secrets.GITHUB_TOKEN }}" \
        -H "X-GitHub-Api-Version: 2022-11-28" \
        "https://api.github.com/repos/newren/git-filter-repo/releases/latest")
      LATEST_TAG=$(echo "${RESPONSE}" | \
        grep '"tag_name"' | \
        sed 's/.*"tag_name"[[:space:]]*:[[:space:]]*"\([^"]*\)".*/\1/' | \
        head -1)
      if [[ -z "${LATEST_TAG}" ]]; then
        echo "ERROR: Could not parse tag_name from GitHub API response" >&2
        echo "Response: ${RESPONSE}" >&2
        exit 1
      fi
      if ! echo "${LATEST_TAG}" | grep -qE '^v[0-9]+\.[0-9]+\.[0-9]+$'; then
        echo "ERROR: Latest tag '${LATEST_TAG}' does not match expected format vX.Y.Z" >&2
        exit 1
      fi
      echo "latest_tag=${LATEST_TAG}" >> "$GITHUB_OUTPUT"
      echo "Latest upstream release: ${LATEST_TAG}"
      ```
    - Step 3 — Get current built version (id: current):
      ```bash
      set -euo pipefail
      CURRENT_RELEASE_TAG=$(grep -E '^RELEASE_TAG=' \
        plugin/.git-filter-repo-config/release.conf | \
        sed 's/^RELEASE_TAG="\(.*\)"$/\1/' | head -1)
      if [[ -z "${CURRENT_RELEASE_TAG}" ]]; then
        echo "ERROR: RELEASE_TAG not found in release.conf" >&2
        exit 1
      fi
      CURRENT_TAG="${CURRENT_RELEASE_TAG#git-filter-repo-}"
      echo "current_tag=${CURRENT_TAG}" >> "$GITHUB_OUTPUT"
      echo "Current built version: ${CURRENT_TAG}"
      ```
    - Step 4 — Check if update needed (id: check):
      ```bash
      set -euo pipefail
      LATEST="${{ steps.upstream.outputs.latest_tag }}"
      CURRENT="${{ steps.current.outputs.current_tag }}"
      if [[ "${LATEST}" == "${CURRENT}" ]]; then
        echo "update_needed=false" >> "$GITHUB_OUTPUT"
        echo "Already up to date (${CURRENT}). No update needed."
      else
        echo "update_needed=true" >> "$GITHUB_OUTPUT"
        echo "Update available: ${CURRENT} -> ${LATEST}"
      fi
      ```
    - Step 5 — Download new source (id: source, `if: steps.check.outputs.update_needed == 'true'`):
      ```bash
      set -euo pipefail
      LATEST="${{ steps.upstream.outputs.latest_tag }}"
      SOURCE_URL="https://raw.githubusercontent.com/newren/git-filter-repo/${LATEST}/git-filter-repo"
      echo "Downloading git-filter-repo source from ${SOURCE_URL}..."
      if ! curl -fsSL --max-time 60 -o git-filter-repo.py "${SOURCE_URL}"; then
        echo "ERROR: Failed to download git-filter-repo source for ${LATEST}" >&2
        exit 1
      fi
      SOURCE_SHA256=$(sha256sum git-filter-repo.py | awk '{print $1}')
      echo "source_sha256=${SOURCE_SHA256}" >> "$GITHUB_OUTPUT"
      echo "SOURCE_SHA256: ${SOURCE_SHA256}"
      ```
    - Step 6 — Update release.conf (`if: steps.check.outputs.update_needed == 'true'`):
      ```bash
      set -euo pipefail
      LATEST="${{ steps.upstream.outputs.latest_tag }}"
      SOURCE_SHA256="${{ steps.source.outputs.source_sha256 }}"
      CONF="plugin/.git-filter-repo-config/release.conf"
      sed -i "s|^RELEASE_TAG=.*|RELEASE_TAG=\"git-filter-repo-${LATEST}\"|" "${CONF}"
      sed -i "s|^SOURCE_SHA256=.*|SOURCE_SHA256=\"${SOURCE_SHA256}\"|" "${CONF}"
      echo "Updated ${CONF}:"
      grep -E '^(RELEASE_TAG|SOURCE_SHA256)=' "${CONF}"
      ```
    - Step 7 — Configure git (`if: steps.check.outputs.update_needed == 'true'`):
      ```bash
      git config user.name "github-actions[bot]"
      git config user.email "41898282+github-actions[bot]@users.noreply.github.com"
      ```
    - Step 8 — Commit and push release.conf update (`if: steps.check.outputs.update_needed == 'true'`):
      ```bash
      set -euo pipefail
      LATEST="${{ steps.upstream.outputs.latest_tag }}"
      git add plugin/.git-filter-repo-config/release.conf
      git commit -m "config: update git-filter-repo source reference to ${LATEST}"
      git push
      ```
    - Step 9 — Trigger build workflow (`if: steps.check.outputs.update_needed == 'true'`):
      ```bash
      set -euo pipefail
      LATEST="${{ steps.upstream.outputs.latest_tag }}"
      gh workflow run build-git-filter-repo.yml \
        -f version="${LATEST}" \
        --ref "${GITHUB_REF_NAME}"
      echo "Triggered build-git-filter-repo.yml for version ${LATEST} on ref ${GITHUB_REF_NAME}"
      ```
      With env: `GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}`

- Update `.github/workflows/build-git-filter-repo.yml` header comment block (lines 7-9) to document
  the automated trigger alongside the existing manual trigger instructions. Replace:
